   ```rust
   pub async fn run_sas_verify(server_url: &str, role: &str) -> Result<(), Box<dyn std::error::Error>>
   ```

   Dispatches to `run_alice(server_url)` or `run_bob(server_url)`.

   **Both alice and bob** need a verification event handler registered before starting the SAS flow. Per D-12 and research Pattern 5, both clients auto-accept verification requests and auto-confirm emoji matches.

   Register a verification event handler on the client (per research Pattern 5):
   ```rust
   // Auto-accept and auto-confirm SAS verification
   client.add_event_handler(|ev: ToDeviceKeyVerificationRequestEvent, client: Client| async move {
       let request = client.encryption()
           .get_verification_request(&ev.sender, &ev.content.transaction_id)
           .await;
       if let Some(request) = request {
           request.accept().await.ok();
           if let Some(sas) = request.start_sas().await.ok().flatten() {
               sas.accept().await.ok();
               let mut stream = sas.changes();
               while let Some(state) = stream.next().await {
                   match state {
                       SasState::KeysExchanged { .. } => {
                           sas.confirm().await.ok();
                       }
                       SasState::Done { .. } => break,
                       SasState::Cancelled(_) => break,
                       _ => {}
                   }
               }
           }
       }
   });
   ```

   Note: The exact import paths and event types depend on matrix-sdk 0.16. The handler may need to use `matrix_sdk::ruma::events::key::verification::*` types. Check the actual matrix-sdk API during implementation -- the research example shows the general pattern but types may differ. Key types to explore:
