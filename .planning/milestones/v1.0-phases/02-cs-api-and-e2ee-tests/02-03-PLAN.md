1. Implement `tests/shadow/src/scenarios/e2ee_msg.rs` with entry point:

   ```rust
   pub async fn run_e2ee_messaging(server_url: &str, role: &str) -> Result<(), Box<dyn std::error::Error>>
   ```

   Dispatches to `run_alice(server_url)` or `run_bob(server_url)`.

   **Alice flow** (per D-11 recommendation: SDK natural sync, research Pattern 4):
   - `create_sdk_client(server_url).await?`
   - `register_with_token(&client, "alice", DEFAULT_PASSWORD, REGISTRATION_TOKEN).await?`
   - `login_user(&client, "alice", DEFAULT_PASSWORD).await?`
   - E2EE-01: First sync uploads device keys automatically:
     `client.sync_once(SyncSettings::default()).await?;`
     `eprintln!("alice: device keys uploaded (first sync)");`
   - Create room with alias `#e2ee-room:tuwunel-server`:
     ```rust
     let mut req = CreateRoomRequest::new();
     req.room_alias_name = Some("e2ee-room".to_owned());
     let room = client.create_room(req).await?;
     ```
   - Enable encryption on the room:
     `room.enable_encryption().await?;`
     `eprintln!("alice: room encryption enabled");`
   - Sync to distribute room keys:
     `client.sync_once(SyncSettings::default()).await?;`
   - Wait for bob to join: poll room members in a loop (up to 30 retries, 2000ms sleep between). Check `room.joined_members_count() >= 2` or equivalent. This is critical per research Pitfall 4 -- alice must see bob before sending encrypted message so the SDK encrypts for bob's device.
     `eprintln!("alice: bob has joined, sending encrypted message");`
   - Sync again to get bob's device keys:
     `client.sync_once(SyncSettings::default()).await?;`
   - E2EE-03: Send encrypted message (SDK handles Megolm transparently):
     ```rust
     room.send(RoomMessageEventContent::text_plain("encrypted secret from alice")).await?;
     eprintln!("alice: encrypted message sent");
     ```
   - `eprintln!("alice: e2ee-messaging scenario complete");`

   **Bob flow:**
   - `create_sdk_client(server_url).await?`
   - `register_with_token(&client, "bob", DEFAULT_PASSWORD, REGISTRATION_TOKEN).await?`
   - `login_user(&client, "bob", DEFAULT_PASSWORD).await?`
   - E2EE-01: First sync uploads device keys:
     `client.sync_once(SyncSettings::default()).await?;`
     `eprintln!("bob: device keys uploaded (first sync)");`
   - E2EE-02: Join encrypted room by alias with retry (same pattern as cs_api bob):
     Join `#e2ee-room:tuwunel-server` with retry loop (30 retries, 1000ms).
     `eprintln!("bob: joined e2ee room");`
   - Sync to receive room state and trigger key claim (SDK handles OTK claim transparently):
     `client.sync_once(SyncSettings::default()).await?;`
     `eprintln!("bob: synced after join (key claim completed)");`
   - Poll for encrypted message: sync in a loop (up to 30 retries, 2000ms) checking for a timeline event containing "encrypted secret from alice". Access timeline events from the sync response or room timeline.
   - When found: `eprintln!("bob: received decrypted message: encrypted secret from alice");`
   - `eprintln!("bob: e2ee-messaging scenario complete");`

   E2EE-04 compliance: All timing uses `tokio::time::sleep` (Shadow-compatible). No wall-clock sleeps. No retry with exponential backoff -- just fixed interval polling under simulated time. The SDK's sync_once drives the key exchange state machine forward deterministically.

   Per D-10: use in-memory crypto store (default when no store is configured in Client::builder). SQLite store support can be added later as a CLI flag.
